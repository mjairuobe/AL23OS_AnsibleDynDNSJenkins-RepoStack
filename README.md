# Ansible DynDNS / Jenkins DevOps Stack

Ansible playbook to provision a **single host** (intended for **Amazon Linux** on EC2 or similar) with a small **CI/CD–oriented** stack: Jenkins behind **HTTPS**, **Docker** and **Dynamic DNS** (No-IP), and supporting tooling.

## What this stack deploys

| Component | Role |
|-----------|------|
| **Swap** | On Amazon Linux, ensures at least a configurable swap file size (see `vars.yaml`) if total swap is below that threshold. |
| **Docker** | Docker engine from distro packages; `jenkins` user added to the `docker` group. |
| **Docker Compose** | Standalone `docker-compose` binary installed from GitHub releases if missing. |
| **Jenkins** | Installed from the official Jenkins RPM repo; **Java 21** (Amazon Corretto headless). Setup wizard disabled in favour of **Configuration as Code (JCasC)**. |
| **Jenkins CasC** | Main definition in `data/config.yaml` (copied to the controller). Admin password and public URL are supplied via `/etc/environment` (`JENKINS_*`) for CasC variable substitution. |
| **Nginx** | Terminates **TLS** on port 443 and reverse-proxies to Jenkins on localhost (`jenkins_http_port`, default 8080). |
| **SSL** | Your **full chain** and **private key** are copied from the control machine into paths defined in `vars.yaml` for Nginx and file permissions. |
| **No-IP** | `noip2` client (SPAL repo), systemd unit and local config deployed from `data/`. |
| **Jenkins plugins & credentials** | Core plugins (CasC, Git, Job DSL, workflow, credentials, etc.) and **Jenkins credentials** for GitHub, Docker Hub, OpenAI, and MongoDB (for use in pipelines / Docker Compose)—installed via Ansible using `community.general` modules. |
| **Git** | Installed on the node for Jenkins/agents. |

**Note:** A **`mongo_port`** variable exists in `vars.yaml` for alignment with application/docker-compose usage; the current playbook does **not** install MongoDB on the server. Run MongoDB separately (e.g. container or managed service) if your pipelines need it.

## Prerequisites

### 1. Configure all YAML and variables before you run

The playbook loads **`vars/vars.yaml`** and **`vars/secrets.yaml`** from `site.yaml`. There is **no** safe default for production: every placeholder must become **your** environment, using the **`.example` files as the checklist**, not as runtime files.

### How the example files work

| File | Purpose |
|------|--------|
| **`vars/vars.yaml.example`** | Lists **every non-secret variable** (URLs, hostnames, paths, ports, credential IDs, swap size, …). |
| **`vars/secrets.yaml.example`** | Lists **every secret** (Jenkins admin password, tokens, registry/API keys, DB passwords). |

Copy the example to the real filename (only once per machine or clone):

   ```bash
   cp vars/vars.yaml.example vars/vars.yaml
   cp vars/secrets.yaml.example vars/secrets.yaml
   ```
Then add your values to the vars.yaml and secrets.yaml.

### 2. No-IP / DynDNS configuration from your provider

The playbook installs the **No-IP** client (`noip2`) and pushes a **local configuration file** to the server. That file is **not** generated by Ansible: it must reflect what **your Dynamic DNS provider** (here: **No-IP**) expects after you register an account and create a hostname.

**Why this matters:** No-IP (and similar providers) tie updates to **your account credentials** and **hostname(s)**. The `noip2` program reads a config file (in this project: paths are wired via `vars.yaml`, source file under `data/`, often `no-ip2.conf`) that typically contains login and hostname information in the **format documented by No-IP**. If you ship an empty template or placeholder text, the daemon may fail, point to the wrong host, or leak generic examples from the internet.

**What you should do:**

1. **Create an account** with your DynDNS provider (e.g. No-IP) and define the **hostname** that should track your server’s public IP (e.g. `yourname.ddns.net`).
2. **Generate or download the client configuration** the provider describes for `noip2` (or follow their wizard / dashboard instructions). Often this means producing a **real `no-ip2.conf`** (or equivalent) containing **your** username, password or token, and **your** hostname(s)—exact syntax is defined by No-IP’s documentation, not by this repo.
3. Place that content in **`data/no-ip2.conf`** on the Ansible control machine (or the path you set in `vars.yaml` as `noip_config_src`). Use **`data/no-ip2.conf.example`** only as a **shape hint**; replace it with **provider-supplied or provider-documented values**. Do **not** commit real credentials; `.gitignore` normally ignores sensitive copies.
4. Ensure **`nginx_server_name`**, **`jenkins_url`**, and your **TLS certificate** match the **same public hostname** you use in DNS so browsers and Jenkins agree on the URL.

If you skip this step, Ansible may still run, but **Dynamic DNS will not work** until a valid provider-owned configuration is in place.

### 3. TLS certificate (required before running the playbook)

The playbook expects **existing certificate material** on the Ansible control node:

- **`fullchain.pem`** (or equivalent) — full certificate chain for Nginx.
- **`privkey.pem`** — private key for the same host name.

**You must obtain these before the run**, for example with **Let’s Encrypt** (e.g. `certbot` on another machine, then copy files here).

Without valid files at the configured `ssl_cert_src` / `ssl_key_src`, the `copy` tasks will fail or Nginx/Jenkins HTTPS will not work correctly.

### 4. Ansible on the control node

- Ansible 2.10+ recommended (FQCN-style builtin modules and collections).
- Install the **community.general** collection (used for `jenkins_plugin` and `jenkins_credential`):

  ```bash
  ansible-galaxy collection install community.general
  ```

## Running the playbook

From the repository root:

```bash
ansible-playbook -i inventory.ini site.yaml
```
